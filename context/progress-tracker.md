## Architecture Decisions

- `context/ui-context.md`'s theme table only gave example Tailwind utility names (`bg-base`, `text-copy-primary`, etc.) without a full mapping for every token. Resolved by extending that table with an explicit "Tailwind Utility" column for every CSS variable, following the naming pattern implied by the given examples (e.g. `border-*` variables â†’ `border-surface-border[-subtle]`, `text-*` variables â†’ `text-copy-*`, state colors â†’ `text-*`/`bg-*` pairs).
- App-level components use the custom tokens documented in `ui-context.md` (`bg-base`, `text-copy-primary`, ...). The shadcn tokens (`bg-primary`, `text-foreground`, `border-border`, ...) remain reserved for use inside `components/ui/*`, both point at the same dark palette so they render identically.
- This project's Next.js version (16.2.10) renamed the `middleware.ts` file convention to `proxy.ts` (functionally identical â€” same `NextMiddleware` type, same execution model, just a new file name and export name). `@clerk/nextjs`'s `clerkMiddleware()` helper still returns a `NextMiddleware`-compatible function, so it default-exports cleanly from `proxy.ts` with no adapter needed.
- Prisma 7 breaking changes from what training data assumes: connection URLs live in `prisma.config.ts`, not in the `datasource` block of `schema.prisma`; `PrismaClient` requires either `adapter` or `accelerateUrl` in its constructor (`new PrismaClient()` with no args throws); the generator is `provider = "prisma-client"` (not `prisma-client-js`) with a mandatory `output` path â€” it no longer generates into `node_modules`. Reference: `.agents/skills/prisma-upgrade-v7/` and sibling `prisma-*` skills (not yet wired into the `Skill` tool's registry, but readable directly as markdown under `.agents/skills/`).
- Multi-file Prisma schema (`prisma/schema.prisma` for generator/datasource, `prisma/models/*.prisma` for models) works out of the box in this project's Prisma version once `prisma.config.ts`'s `schema` field points at the `prisma/` directory rather than a single file â€” confirmed via `npx prisma validate`.
- The project's Liveblocks room id is just its Prisma `id` (the server-generated cuid) â€” there is no separate `roomId` column and none was added. `07-wire-editor-home.md` asked for the create hook to generate a slug+suffix "room id" client-side before calling the API, but `06-project-apis.md` forbids client-supplied ids ("use the schema's existing ID strategy"); resolved (user-confirmed) by treating that generated string as a cosmetic preview only in the Create dialog, and using the real `id` the API returns for both navigation and (whenever Liveblocks is actually wired up) the room id.
- Project-list data for the editor sidebar is fetched in `app/editor/layout.tsx`, not `app/editor/page.tsx`, even though `07-wire-editor-home.md` says "the editor home page is a server component." The sidebar is mounted by the shared layout (so it also needs to be populated for `/editor/[roomId]`, not just `/editor`), and `layout.tsx` was already a server component with no client-only content of its own â€” so it's the natural single fetch point. `ProjectDialogsProvider` no longer keeps its own mutated copy of the project list; it just forwards the layout's server-fetched props through context, and mutations rely on `router.refresh()`/`router.push()` to pull fresh data rather than local array splicing.
- `10-liveblocks-setup.md`'s auth route reads `room` (not `roomId`) from the POST body because that's the literal field name `@liveblocks/client` sends when `authEndpoint` is a plain string URL (verified by reading `@liveblocks/core`'s compiled source rather than assuming from memory, per `AGENTS.md`'s warning that this Next.js/package setup may differ from training data).
- `lib/liveblocks.ts` exports a `getLiveblocksClient()` function instead of an eagerly-constructed module-level client (unlike `lib/prisma.ts`'s eager singleton). `@liveblocks/node`'s `Liveblocks` constructor synchronously validates the secret key format and throws if it doesn't start with `sk_`; Next.js's build step evaluates route modules during "Collecting page data" even for a POST-only handler, so an eager `new Liveblocks(...)` at module scope failed `npm run build` outright whenever a real key isn't set yet (confirmed by reproducing the failure with the current empty placeholder key). Deferring construction until the handler actually runs sidesteps this; the `globalThis` caching itself still mirrors `lib/prisma.ts`'s pattern.
- `11-base-canvas.md`'s "error fallback for Liveblocks connection issues" is implemented with `useErrorListener` (via `CanvasConnectionGuard` in `components/editor/canvas-room.tsx`), not a React `ErrorBoundary` around `ClientSideSuspense` â€” the two are not interchangeable here. Liveblocks' auth-retry state machine only calls `fireErrorEvent` (which `useErrorListener` receives) for a fixed set of non-retryable HTTP status codes (`400/401/403/404/405/410/...`); everything else, including a `500`, is treated as transient and retried with backoff forever, without ever throwing into the render tree. A plain `ErrorBoundary` therefore only protects against genuine synchronous render throws (kept as `CanvasErrorBoundary`, still wrapping `ClientSideSuspense` as defense-in-depth), while `useErrorListener` is what actually surfaces the realistic "user lost access to this room" (`403`) case. Confirmed by reading `@liveblocks/core`'s compiled source and by direct testing (a `403` triggers the fallback; a `500`, correctly, keeps retrying/loading).
- `12-shape-panel.md`'s drag-and-drop node creation dispatches through `useLiveblocksFlow`'s own `onNodesChange` with a React-Flow-native `{ type: "add", item }` change, rather than any Liveblocks-specific "add node" call â€” `@liveblocks/react-flow` ships a server-only `mutateFlow`/`flow.addNode()` API (`@liveblocks/react-flow/node`, for webhook/backend use), but the client-side hook's `onNodesChange` is the only mutation surface exposed to a React component, and it recognizes the same `NodeChange` union (`add`/`position`/`select`/`remove`/`replace`/`dimensions`) that `@xyflow/react`'s own `applyNodeChanges` does â€” confirmed by reading the compiled source, not assumed from training data, per `AGENTS.md`.
- `components/editor/canvas.tsx` now wraps its React Flow instance in an explicit `<ReactFlowProvider>` (a `Canvas` â†’ `CanvasInner` split), which wasn't needed in `11` since nothing used `useReactFlow()` yet. `useReactFlow()` (needed for `screenToFlowPosition`, the documented way to convert a drop event's `clientX`/`clientY` into canvas coordinates) only works in a descendant of `<ReactFlow>`'s own internally-rendered provider â€” calling it in the same component function that returns `<ReactFlow>...</ReactFlow>` throws, since that provider doesn't exist yet during that component's own render.
- `@liveblocks/react-flow`'s `useLiveblocksFlow` hook's `onNodesChange`/`onEdgesChange` mutations do **not** implement the `"remove"` `NodeChange`/`EdgeChange` case in the version installed here (`node_modules/@liveblocks/react-flow/dist/lib/flow.js`'s `applyNodeChanges`/`applyEdgeChanges` both have a literal `case "remove": break;`) - dispatching a remove through them compiles, typechecks, and silently does nothing at runtime. Programmatic node/edge deletion (`18-starter-template.md`'s "clear the canvas before importing") must go through the hook's separate fourth return value, `onDelete({ nodes, edges })` (full objects, matching `@xyflow/react`'s `OnDelete` type - normally wired to `<ReactFlow onDelete={...}>`'s built-in delete-key handling, but callable directly too), which deletes straight from the underlying `LiveMap` instead of routing through the no-op change reducer. Found via a subagent smoke test, confirmed by reading the library's compiled source rather than assumed, per `AGENTS.md`.
- `08-editor-workspace-shell.md` renamed the workspace route's dynamic segment from `[projectId]` to `[roomId]` â€” cosmetic/naming only, still the same Prisma `id` (see the `07` decision above on room-id = Prisma id). Rather than build a second, page-specific navbar/sidebar/right-panel shell for `/editor/[roomId]`, the existing shared `EditorShell` (mounted once in `app/editor/layout.tsx` for every `/editor/**` route) was made room-aware via `useParams` â€” it now conditionally shows the project name/Share/AI-toggle navbar actions and the right AI placeholder panel only when a `roomId` is present in the URL, leaving `/editor` home's chrome unchanged. This avoids double-mounting a navbar or sidebar, at the cost of `EditorShell` needing to know about room-specific UI it otherwise wouldn't.
