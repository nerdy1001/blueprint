## Architecture Decisions

- `context/ui-context.md`'s theme table only gave example Tailwind utility names (`bg-base`, `text-copy-primary`, etc.) without a full mapping for every token. Resolved by extending that table with an explicit "Tailwind Utility" column for every CSS variable, following the naming pattern implied by the given examples (e.g. `border-*` variables → `border-surface-border[-subtle]`, `text-*` variables → `text-copy-*`, state colors → `text-*`/`bg-*` pairs).
- App-level components use the custom tokens documented in `ui-context.md` (`bg-base`, `text-copy-primary`, ...). The shadcn tokens (`bg-primary`, `text-foreground`, `border-border`, ...) remain reserved for use inside `components/ui/*`, both point at the same dark palette so they render identically.
- This project's Next.js version (16.2.10) renamed the `middleware.ts` file convention to `proxy.ts` (functionally identical — same `NextMiddleware` type, same execution model, just a new file name and export name). `@clerk/nextjs`'s `clerkMiddleware()` helper still returns a `NextMiddleware`-compatible function, so it default-exports cleanly from `proxy.ts` with no adapter needed.
- Prisma 7 breaking changes from what training data assumes: connection URLs live in `prisma.config.ts`, not in the `datasource` block of `schema.prisma`; `PrismaClient` requires either `adapter` or `accelerateUrl` in its constructor (`new PrismaClient()` with no args throws); the generator is `provider = "prisma-client"` (not `prisma-client-js`) with a mandatory `output` path — it no longer generates into `node_modules`. Reference: `.agents/skills/prisma-upgrade-v7/` and sibling `prisma-*` skills (not yet wired into the `Skill` tool's registry, but readable directly as markdown under `.agents/skills/`).
- Multi-file Prisma schema (`prisma/schema.prisma` for generator/datasource, `prisma/models/*.prisma` for models) works out of the box in this project's Prisma version once `prisma.config.ts`'s `schema` field points at the `prisma/` directory rather than a single file — confirmed via `npx prisma validate`.
- The project's Liveblocks room id is just its Prisma `id` (the server-generated cuid) — there is no separate `roomId` column and none was added. `07-wire-editor-home.md` asked for the create hook to generate a slug+suffix "room id" client-side before calling the API, but `06-project-apis.md` forbids client-supplied ids ("use the schema's existing ID strategy"); resolved (user-confirmed) by treating that generated string as a cosmetic preview only in the Create dialog, and using the real `id` the API returns for both navigation and (whenever Liveblocks is actually wired up) the room id.
- Project-list data for the editor sidebar is fetched in `app/editor/layout.tsx`, not `app/editor/page.tsx`, even though `07-wire-editor-home.md` says "the editor home page is a server component." The sidebar is mounted by the shared layout (so it also needs to be populated for `/editor/[roomId]`, not just `/editor`), and `layout.tsx` was already a server component with no client-only content of its own — so it's the natural single fetch point. `ProjectDialogsProvider` no longer keeps its own mutated copy of the project list; it just forwards the layout's server-fetched props through context, and mutations rely on `router.refresh()`/`router.push()` to pull fresh data rather than local array splicing.
- `08-editor-workspace-shell.md` renamed the workspace route's dynamic segment from `[projectId]` to `[roomId]` — cosmetic/naming only, still the same Prisma `id` (see the `07` decision above on room-id = Prisma id). Rather than build a second, page-specific navbar/sidebar/right-panel shell for `/editor/[roomId]`, the existing shared `EditorShell` (mounted once in `app/editor/layout.tsx` for every `/editor/**` route) was made room-aware via `useParams` — it now conditionally shows the project name/Share/AI-toggle navbar actions and the right AI placeholder panel only when a `roomId` is present in the URL, leaving `/editor` home's chrome unchanged. This avoids double-mounting a navbar or sidebar, at the cost of `EditorShell` needing to know about room-specific UI it otherwise wouldn't.

## Session Notes

- shadcn CLI (v4.13.0) uses a new preset system; installed with the "Nova" preset (`Lucide / Geist`), which matches the project's existing icon/font choices, and the `radix` base library (Radix UI primitives via the unified `radix-ui` package) rather than the newer non-Radix `base` option.
- Do not modify `components/ui/*` — per `context/ai-workflow-rules.md`, project-specific styling belongs in app-level components instead.
- Next.js app router treats any `app/` folder prefixed with `_` as a private, non-routable segment — a temporary smoke-test route must use a plain folder name (e.g. `app/smoke-test/`) or it 404s.
- Gotcha: since `--color-base` is defined as a theme token in `globals.css`, Tailwind v4 auto-generates `bg-base` *and* `text-base` as color utilities — meaning `text-base` here is a text **color** (page background color), not Tailwind's default font-size utility. Don't use `text-base` for sizing; it silently collides with any text-color utility on the same element.
- Now that Clerk auth guards every route via `proxy.ts`, smoke-testing anything under `/editor` needs a temporary bypass: add a throwaway route (e.g. `app/smoke-test-04/`) and temporarily add its matcher (e.g. `"/smoke-test-04(.*)"`) to the `isPublicRoute` list in `proxy.ts`, then revert both after testing. Confirm the revert with `git status`/`git diff proxy.ts` before finishing.
- Gotcha: with Clerk dev-instance keys, direct `curl`/HTTP-client requests to any route behind `proxy.ts` (including `/api/*`) get a `307` redirect to `/sign-in` with header `x-clerk-auth-reason: dev-browser-missing`, regardless of method or real auth state — Clerk requires a browser-established `__clerk_db_jwt` handshake (set client-side by `clerk-js` on first page load) before it will even evaluate signed-in/signed-out status. There's no way to complete this handshake from curl. To exercise real 401/signed-out API behavior, use a headless browser (already carries the cookie from loading any page first) rather than curl.
- Gotcha: `npx playwright` doesn't add a `playwright` package to this project's `node_modules` — it runs from npm's own npx cache (`~/AppData/Local/npm-cache/_npx/<hash>/node_modules/playwright`). A standalone `node script.mjs` that does `import { chromium } from "playwright"` only resolves if the script is colocated in that same cache directory (find it via the cache path, `cp` the script there, then run `node script.mjs` from that directory).
- To smoke-test a feature that needs a *real* authenticated session (not just an unauthenticated redirect check), create a Clerk dev-instance test user via the Backend API using `CLERK_SECRET_KEY` from `.env.local` (`POST https://api.clerk.com/v1/users` with an `+clerk_test@` email and `skip_password_checks: true`), then drive the actual `/sign-in` UI with Playwright. Two OTP-style prompts can appear, both accept the fixed dev-mode code `424242`: the normal email-code step (if password sign-in isn't used/available), and — even after a successful password sign-in — a separate "check your device"/`client-trust` step that Clerk inserts for a browser it hasn't seen before (Playwright's context always looks new). Wait for navigation to settle after filling that code before asserting on the destination URL; it does a couple of internal redirects (`/sign-in/client-trust` → `/` → the app's real post-auth redirect target). Delete the test user via `DELETE /v1/users/<id>` on the same Backend API when done — it isn't cleaned up automatically. The Clerk CLI itself (`npx clerk@latest`) requires an interactive browser login (`clerk auth login`) and cannot authenticate non-interactively in this environment, so it's not viable for this — go straight to the Backend API with the app's existing secret key instead. For a two-user scenario (e.g. owner vs. collaborator), use two separate `browser.newContext()`s in the same Playwright script rather than signing in/out serially in one context — avoids cookie/session cleanup between the two and lets both be logged in "simultaneously" for cross-checks.
- Gotcha (confirmed as a real, user-reported bug, not just theoretical — and it recurred a second time): `--color-base` being a `@theme` color token means Tailwind v4 repurposes `text-base`/`bg-base` as *that color* (near-black, `#080809`) instead of the framework-default font-size meaning. shadcn's generated `components/ui/input.tsx`, `textarea.tsx`, and `dialog.tsx` (`DialogTitle`) all shipped with a stray `text-base` class from the original templates (meant as font-sizing). `tailwind-merge` (used in `lib/utils.ts`'s `cn()`) doesn't know about this project's custom color tokens, so it doesn't dedupe `text-base` against an explicit `text-copy-primary` override placed later in the same class string — both classes end up in the DOM, and plain CSS cascade (source order in the generated stylesheet, not class-attribute order) decided the color, which happened to make the text render almost invisible against the dark background. Fixed in all three files (`input.tsx`/`textarea.tsx` during `09`'s initial build, `dialog.tsx` during the same feature's later visual-redesign pass). `components/ui/card.tsx` has the identical `text-base` pattern (line ~41, card title) and has **not** been checked/fixed — it just hasn't been hit yet since `Card` isn't used in an app-level component with an explicit color override the way dialogs/inputs are. If `Card` gets used for something with dark-on-dark text, check this first. General rule going forward: check computed text color, not just presence/absence of a color-utility class in the source, when verifying dark-theme text legibility on any `components/ui/*` primitive that has a bare `text-base` in its default className.
- Gotcha: `eslint-config-next`'s React Compiler rules include `react-hooks/set-state-in-effect`, which errors (not warns) on *any* direct `setState` call in a `useEffect` body — including the extremely common "set a loading flag, then fetch" pattern. There's no targeted escape hatch that keeps the pattern; the fix used here (`hooks/use-share-dialog.ts`) was to not use an effect at all for "fetch when a dialog opens" — instead expose a plain async `load()` function and call it directly from the real click handler that flips the dialog's `open` state (in `EditorShell`, one level up), since that handler is a genuine user-event callback rather than an effect. Worth checking for this pattern before reaching for `useEffect` + `fetch` on any future on-demand-loaded UI (dialogs, popovers, tabs).
